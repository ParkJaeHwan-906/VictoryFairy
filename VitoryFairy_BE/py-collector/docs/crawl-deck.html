02 한눈에 · 소스 → 싱크

어디서 긁어 어디에 쌓는가

네이버 스포츠 API

일정·결과·문자중계·박스스코어(record). 안정적 JSON, 종료 경기도 영구 조회 → 백필 가능.

KBO 공식 사이트

구단별 1군 등록 명단·포지션. 일자별 제출되는 유일한 신뢰 소스.

DCInside · FMKorea

팬 반응·여론. 구단별 갤러리 = 팀 단위 세그먼트.

S3 원본 랜딩존

일정·결과·문자중계 원본 JSON + 커뮤니티 글. 가공 없이 멱등 적재.

MySQL 정규화 저장소

구단·선수 로스터 + 경기 박스스코어. 즉시 질의·조인 가능한 관계형.

→ 성격에 따라 원본 보존(S3) 과 정규화(MySQL) 로 갈린다.

04 데이터 처리 파이프라인

어디까지 가공하나 — 두 가지 모드

원본 보존 schema-on-read

네이버 schedule·result·relay → 응답 byte-for-byte로 S3. 가공 0.

원본 손실 방지·스키마 결합 최소화. 필드 추출은 분석 시점으로 미룸.

적재 시 정규화 schema-on-write

record·로스터·커뮤니티 → 파싱·정형화해 MySQL / RawPost로.

즉시 질의·조인·집계가 필요한 데이터.

원천3 소스
→
브론즈 (현재)S3 원본 JSON · MySQL 정규화 · RawPost(+마스킹)
→
소비Spring읽기전용 엔티티가 MySQL 직접 조회

향후: S3 브론즈 → Silver/Gold 분석계층(집계·Parquet·relay play-by-play 파싱) — 원본 보존돼 소급 처리 가능.

05 정규화 계층 처리

정규화 단계에서 실제로 하는 일

PII 마스킹

댓글 작성자 salt+SHA-256[:12] 익명화(역추적 불가). 본문·제목은 무필터 보존.

식별자 해소

네이버 pcode 종속 제거 → 자체 player_uid 발급, 로스터 kbo_player_id와 이름+팀 유일매칭 링크.

JSON → 관계형

record 중첩 JSON을 5개 평면 테이블로. 이닝 배열은 JSON 컬럼 대신 game_innings 행으로.

시계열 모델링

players(현재상태) + player_registrations(일자 스냅샷) → 특정일 등록현황 재현.

멱등 병합

전 경로 upsert/멱등 키 → 재수집·재백필해도 중복 없이 갱신.
